=== Limitations
:rev: {project-version}

Understanding the limitations of Spring Data LadybugDB is crucial for building reliable applications.

==== Single Writer Constraint

LadybugDB allows only one write transaction at a time.

===== Official Documentation

> "At any point in time, there can be multiple read transactions but only one write transaction"

===== Implications

[cols="1,2"]
|===
| Scenario | Behavior

| Multiple reads
| Run in parallel without blocking

| Single write
| Executes immediately

| Concurrent writes
| Block waiting for write lock

| Write during reads
| Blocks until all reads complete
|===

===== Mitigation Strategies

==== Keep Writes Short

[source,java]
----
@Transactional
public void quickUpdate(String id, int value) {
    // Single, fast operation
    template.execute(
        "MATCH (n:Node {id: $id}) SET n.value = $value",
        Map.of("id", id, "value", value)
    );
}
----

==== Use Connection Pooling

[source,java]
----
@Bean
public PooledConnectionFactory connectionFactory(Database database) {
    GenericObjectPoolConfig<Connection> config = new GenericObjectPoolConfig<>();
    config.setMaxTotal(10);  // Limit concurrent connections
    return new PooledConnectionFactory(database, config);
}
----

==== Consider Read-Only Transactions

[source,java]
----
@Transactional(readOnly = true)
public List<Person> findAllAdults() {
    return template.query("MATCH (n:Person) WHERE n.age >= 18 RETURN n", Person.class);
}
----

==== No Rollback Support

Each command auto-commits immediately. Rollback is not supported.

===== Transaction Manager Behavior

The `LadybugDBTransactionManager` provides:

* Connection binding within transaction scope
* Consistent connection usage for multiple operations

It does NOT provide:

* Atomic transactions
* Rollback capability
* Isolation levels

===== Example

[source,java]
----
@Transactional
public void updateMultiple(List<Person> people) {
    for (Person p : people) {
        template.execute(
            "MATCH (n:Person {name: $name}) SET n.age = $age",
            Map.of("name", p.getName(), "age", p.getAge())
        );
        // Each SET is immediately committed!
    }
    
    throw new RuntimeException("Error!");
    // Previous updates are NOT rolled back
}
----

===== Best Practices

1. Validate inputs before database operations
2. Handle errors gracefully
3. Design for idempotency where possible
4. Use compensating transactions for rollback logic

==== Connection Binding Only

The transaction manager binds a connection to the thread but doesn't use explicit transactions.

===== What Works

[source,java]
----
@Transactional
public void processBatch(List<Item> items) {
    // All operations use the same connection
    for (Item item : items) {
        template.execute(...);
    }
}
----

===== What Doesn't Work

[source,java]
----
@Transactional
public void atomicUpdate() {
    template.execute("CREATE (n:Node {id: 1})");
    template.execute("CREATE (n:Node {id: 2})");
    
    throw new RuntimeException();
    // Node 1 still exists - no rollback
}
----

==== Relationship Direction

Relationships are directional in LadybugDB.

===== Creating Relationships

[source,java]
----
// Creates: Alice -[:KNOWS]-> Bob
repository.createRelation(alice, bob, knows);
----

===== Querying Both Directions

[source,java]
----
// To find bidirectional relationships:
List<Person> allConnections = template.query(
    """
    MATCH (p:Person)-[:KNOWS]-(friend:Person)
    WHERE p.name = $name
    RETURN DISTINCT friend
    """,
    Map.of("name", "Alice"),
    Person.class
);
----

===== Creating Bidirectional Relationships

[source,java]
----
// Create both directions for bidirectional relationship
repository.createRelation(alice, bob, knowsAliceBob);
repository.createRelation(bob, alice, knowsBobAlice);
----

==== Entity Type Constraints

===== Source and Target Must Be Same Type

[source,java]
----
@RelationshipEntity(
    type = "KNOWS",
    nodeType = Person.class,  // Both source and target
    sourceField = "from",
    targetField = "to"
)
public class Knows {
    Person from;
    Person to;
}
----

===== For Different Node Types

Use the template directly:

[source,java]
----
template.execute(
    "MATCH (p:Person {id: $personId}), (c:Company {id: $companyId}) CREATE (p)-[:WORKS_AT]->(c)",
    Map.of("personId", "Alice", "companyId", "Acme")
);
----

==== Query Method Limitations

===== Derived Queries

Query derivation from method names is limited. Prefer explicit `@Query` annotations:

[source,java]
----
// May not work as expected
List<Person> findByAgeGreaterThan(int age);

// Preferred: Explicit query
@Query("MATCH (n:Person) WHERE n.age > $age RETURN n")
List<Person> findByAgeGreaterThan(@Param("age") int age);
----

===== Pagination

No built-in pagination support. Use LIMIT and SKIP:

[source,java]
----
@Query("MATCH (n:Person) RETURN n ORDER BY n.name SKIP $offset LIMIT $limit")
List<Person> findAllPaginated(@Param("offset") int offset, @Param("limit") int limit);
----

===== Sorting

No dynamic sorting. Include ORDER BY in queries:

[source,java]
----
@Query("MATCH (n:Person) RETURN n ORDER BY n.name ASC")
List<Person> findAllOrderedByName();

@Query("MATCH (n:Person) RETURN n ORDER BY n.age DESC")
List<Person> findAllOrderedByAgeDesc();
----

==== Native Library Dependency

LadybugDB requires its native library to be installed.

===== Platform Support

Check https://ladybugdb.com[LadybugDB documentation] for supported platforms.

===== Library Path

The JVM must find the native library:

[source,bash]
----
export LD_LIBRARY_PATH=/path/to/ladybugdb/lib:$LD_LIBRARY_PATH
----

===== Error Handling

[source,java]
----
try {
    Database db = new Database(":memory:");
} catch (UnsatisfiedLinkError e) {
    // Native library not found
    System.err.println("Please install LadybugDB native library");
}
----

==== Memory Considerations

===== Large Result Sets

Use streaming for large results:

[source,java]
----
// Avoid: Loads all into memory
List<BigEntity> all = template.query("MATCH (n) RETURN n", BigEntity.class);

// Better: Stream results
try (Stream<BigEntity> stream = template.stream("MATCH (n) RETURN n", BigEntity.class)) {
    stream.forEach(this::process);
}
----

===== Connection Pool Sizing

Size the pool appropriately:

[source,java]
----
GenericObjectPoolConfig<Connection> config = new GenericObjectPoolConfig<>();
config.setMaxTotal(10);  // Not too high - single writer constraint
config.setMinIdle(2);    // Keep some connections ready
----
